01 — Introducción

¿Cómo está construida la app?

Antes de explicar el deploy, hay que entender las tres partes de la aplicación.

"Esta aplicación tiene tres capas separadas. El frontend hecho en React es lo que el usuario ve en el navegador. El backend hecho en Node.js con Express es el servidor que recibe las peticiones y aplica la lógica de negocio. Y la base de datos en MySQL guarda toda la información permanentemente."


"Cuando alguien se registra en la app, pasan tres cosas: el frontend manda los datos al backend via HTTP, el backend valida y encripta la contraseña, y finalmente guarda el usuario en MySQL."

  • Mostrar la URL del frontend en Vercel funcionando
  • Mostrar el diagrama: Vercel → Railway Backend → Railway MySQL
02 — Base de Datos

MySQL en Railway

Railway nos da una base de datos MySQL completamente configurada en segundos.

"Para el deploy usamos Railway, una plataforma que nos permite hostear aplicaciones y bases de datos de forma gratuita. Primero creamos un proyecto nuevo y agregamos un servicio de MySQL."


"Lo importante aquí es que Railway nos genera automáticamente todas las credenciales — el host, el usuario, la contraseña y el puerto. No tenemos que instalar MySQL ni configurar nada manualmente. Todo queda listo en menos de un minuto."

  • Mostrar Railway → proyecto → servicio MySQL → pestaña Variables
  • Señalar las variables: MYSQLHOST, MYSQLPASSWORD, MYSQLDATABASE
  • Decir: "estas credenciales las vamos a referenciar desde el backend"

Railway usa contenedores Docker para correr MySQL. Cuando creamos el servicio, Railway levanta un contenedor con MySQL ya configurado y nos expone un puerto público para conectarnos. Internamente, los servicios del mismo proyecto se comunican por una red privada usando el hostname mysql.railway.internal.

03 — Servidor

Backend en Railway

Railway lee nuestro repositorio de GitHub y hace el deploy automáticamente.

"Dentro del mismo proyecto de Railway, agregamos un segundo servicio conectando nuestro repositorio de GitHub. Railway detecta automáticamente que es una aplicación Node.js, instala las dependencias y la levanta."


"Como nuestro repositorio tiene dos carpetas — backend y frontend — le decimos a Railway que solo mire la carpeta del backend configurando el Root Directory."

  • Mostrar Railway → servicio DesarrolloWeb → Settings
  • Señalar Root Directory: profesor-ratings-app/backend
  • Mostrar que el deploy dice "Deployment successful"
  • Mostrar los logs: "Conectado a MySQL" y "Migración completada"

Si no configuramos el Root Directory, Railway intenta buildear desde la raíz del repo y no encuentra el package.json del backend. Al apuntarle a la subcarpeta correcta, Railway encuentra el archivo y sabe exactamente qué instalar y cómo iniciar el servidor.

ℹ️ Importante: El backend y MySQL deben estar en el mismo proyecto de Railway. Así pueden comunicarse internamente sin pasar por internet, lo que es más rápido y seguro.
04 — Configuración

Variables de entorno

Las variables de entorno reemplazan el archivo .env en producción.

"En desarrollo usamos un archivo .env en nuestra computadora con contraseñas y configuración. Ese archivo nunca se sube a GitHub por seguridad. En producción, Railway nos permite configurar esas mismas variables directamente en la plataforma."


"Para las credenciales de MySQL usamos referencias — en lugar de copiar y pegar la contraseña, le decimos a Railway que tome el valor directamente del servicio MySQL que está en el mismo proyecto. Si la contraseña cambia algún día, se actualiza automáticamente en todos lados."

  • Mostrar Railway → servicio backend → Variables
  • Señalar que DB_HOST tiene el valor mysql.railway.internal (referencia resuelta)
  • Señalar CORS_ORIGIN con la URL de Vercel

05 — Cambio en el código

El cambio que había que hacer en api.js

Este fue el cambio más importante en el código antes del deploy.

"Cuando revisamos el código del frontend, encontramos que el archivo que maneja todas las peticiones HTTP tenía la URL del backend escrita fija como localhost. Eso funciona perfectamente cuando trabajamos en nuestra computadora, pero en producción localhost no existe."


"El cambio fue simple: en lugar de tener la URL hardcodeada, ahora el código lee una variable de entorno. Si existe esa variable, la usa. Si no existe, cae de vuelta a localhost para desarrollo. Así el mismo código funciona en ambos ambientes."

frontend/src/services/api.js 
// ANTES — URL fija, no funciona en producción ❌
const API_URL = 'http://localhost:5000/api';

// DESPUÉS — Lee variable de entorno, cae a localhost si no existe ✅
const API_URL = process.env.REACT_APP_API_URL || 'http://localhost:5000/api';

En React, las variables de entorno deben empezar con REACT_APP_ para que estén disponibles en el navegador. Es una medida de seguridad de React que evita exponer accidentalmente variables del servidor al cliente. La variable REACT_APP_API_URL la configuramos en Vercel con la URL de Railway.

  • Mostrar el archivo api.js en VS Code con el cambio
  • Mostrar Vercel → Settings → Environment Variables → REACT_APP_API_URL
06 — Automatización

Migración automática de tablas

Las tablas se crean solas cuando el servidor inicia por primera vez.

"Normalmente para crear las tablas en la base de datos necesitaríamos conectarnos con una herramienta como MySQL Workbench o TablePlus y ejecutar el schema manualmente. Eso es un paso extra que puede fallar."


"Para evitar eso, agregamos un script de migración al backend. Cuando el servidor inicia, antes de aceptar cualquier petición, revisa si las tablas existen. Si no existen, las crea automáticamente. Usamos CREATE TABLE IF NOT EXISTS para que no falle si las tablas ya están creadas."

backend/server.js 
const migrate = require('./src/config/migrate');

// Primero crea las tablas, luego inicia el servidor
migrate()
  .then(() => {
    app.listen(PORT, () => {
      console.log(`Servidor en puerto ${PORT}`);
    });
  });
  • Mostrar Railway → backend → Deploy Logs
  • Señalar: "✅ Migración completada — tablas listas"
  • Señalar: "Conectado a la base de datos MySQL"
  • Señalar: "Servidor corriendo en puerto 5000"

El orden importa: primero nos conectamos a MySQL, luego corremos la migración, y solo después levantamos el servidor. Si alguno de esos pasos falla, el proceso termina con error y Railway nos avisa. Esto evita que el servidor quede corriendo sin tablas.

07 — Seguridad

¿Qué es CORS y por qué importa?

CORS es el mecanismo que controla qué dominios pueden hacer peticiones al backend.

"CORS significa Cross-Origin Resource Sharing. Es una medida de seguridad del navegador que bloquea peticiones entre dominios diferentes. Por ejemplo, si nuestro frontend está en Vercel y el backend en Railway, son dominios distintos."


"Si no configuramos CORS, el navegador bloquea todas las peticiones del frontend al backend con un error. La solución es decirle al backend qué dominios están autorizados para hacer peticiones. En este caso, solo permitimos la URL de Vercel."

backend/src/app.js 
app.use(cors({
  origin: process.env.CORS_ORIGIN || 'http://localhost:3000',
  credentials: true
}));

La variable CORS_ORIGIN en Railway tiene el valor https://desarrollo-web-12026-ejemplo.vercel.app. Solo peticiones desde esa URL son aceptadas. Cualquier otro dominio recibe error 403. En desarrollo cae a localhost:3000 automáticamente.

08 — Frontend

Deploy del frontend en Vercel

Vercel está especializado en frontends React y hace el build automáticamente.

"Para el frontend usamos Vercel, que está especializado en aplicaciones React. La ventaja es que el plan gratuito es prácticamente ilimitado para frontends estáticos."


"El proceso es igual que Railway — conectamos el repositorio de GitHub, le decimos que la carpeta del frontend es profesor-ratings-app/frontend, y configuramos la variable de entorno con la URL del backend de Railway. Vercel buildea la app con npm run build y la despliega en una CDN global."

  • Mostrar vercel.com → proyecto → último deployment "Ready"
  • Mostrar Settings → Environment Variables → REACT_APP_API_URL
  • Abrir la URL del frontend en el navegador
✅ Cada vez que hacemos git push al repositorio, Vercel y Railway detectan el cambio automáticamente y hacen un nuevo deploy. No hay que hacer nada manual.
